# `EBS (Elastic Block Store)` :

-   An EBS (Elastic Block Store) Volume is a network drive you can attach to your instances while they run.
-   It allows your instances to persist data, even after their termination
-   They are bound to a specific availability zone
    > Free tier: 30 GB of free EBS storage of type General Purpose (SSD) or
    > Magnetic per month
-   It uses the network to communicate the instance, which means there might be a bit of latency
-   It can be detached from an EC2 instance and attached to another one quickly
-   An EBS Volume in us-east-1a cannot be attached to us-east-1b, To move a volume across, you first need to snapshot it
-   You get billed for all the provisioned capacity, You can increase the capacity of the drive over time

**EBS – Delete on Termination attribute**

-   Controls the EBS behaviour when an EC2 instance terminates
-   By default, the root EBS volume is deleted (attribute enabled)
-   By default, any other attached EBS volume is not deleted (attribute disabled), This can be controlled by the AWS console / AWS CLI

**EBS Snapshots :**

-   Make a backup (snapshot) of your EBS volume at a point in time
-   Not necessary to detach volume to do snapshot, but recommended
-   Can copy snapshots across AZ or Region
-   EBS Snapshots Features :
    -   EBS Snapshot Archive :
        -   Move a Snapshot to an ”archive tier” that is 75% cheaper
        -   Takes within 24 to 72 hours for restoring the archive
    -   Recycle Bin for EBS Snapshots :
        -   Setup rules to retain deleted snapshots so you can recover them after an accidental deletion
        -   Specify retention (from 1 day to 1 year)
    -   Fast Snapshot Restore (FSR) :
        -   Force full initialization of snapshot to have no latency on the first use ($$$)

-   **General Purpose SSD** : Used for System boot volumes, Virtual desktops, Development and test environments.
    -   gp2
    -   gp3
-   **Provisioned IOPS (PIOPS) SSD** : Used for Critical business applications with sustained IOPS performance. Or applications that need more than 16,000 IOPS. Great for databases workloads (sensitive to storage perf and consistency).

# `Amazon EFS(Elastic File System) `

-   It's a managed NFS (network file system) that can be mounted on many EC2
-   EFS works with EC2 instances in _multi-AZ_
-   Highly available, scalable, expensive, pay per use
-   Use cases: content management, web serving, data sharing, Wordpress
-   Uses NFSv4.1 protocol
-   Uses security group to control access to EFS
-   Compatible with Linux based AMI (not Windows)
-   Encryption at rest using KMS
-   POSIX file system (~Linux) that has a standard file API
-   File system scales automatically, pay-per-use, no capacity planning!
-   **EFS – Performance Classes**:
    -   EFS Scale :
        -   1000s of concurrent NFS clients, 10 GB+ /s throughput
        -   Grow to Petabyte-scale network file system, automatically
    -   Performance Mode (set at EFS creation time) :
        -   General Purpose (default) – latency-sensitive use cases (web server, CMS, etc…)
        -   Max I/O – higher latency, throughput, highly parallel (big data, media processing)
    -   Throughput Mode :
        -   Bursting – 1 TB = 50MiB/s + burst of up to 100MiB/s
        -   Provisioned – set your throughput regardless of storage size, ex: 1 GiB/s for 1 TB storage
        -   Elastic – automatically scales throughput up or down based on your workloads
            -   Up to 3GiB/s for reads and 1GiB/s for writes
            -   Used for unpredictable workloads
-   **EFS – Storage Classes**:
    -   Storage Tiers (lifecycle management feature – move file after N days) :
        -   Standard: for frequently accessed files
        -   Infrequent access (EFS-IA): cost to retrieve files, lower price to store.
        -   Archive: rarely accessed data (few times each year), 50% cheaper
        -   Implement lifecycle policies to move files between storage tiers
    -   Availability and durability :
        -   Standard: Multi-AZ, great for prod
        -   One Zone: One AZ, great for dev, backup enabled by default, compatible with IA (EFS One Zone-IA)

# `Elastic Load Balancer`

Load Balances are servers that forward traffic to multiple
servers (e.g., EC2 instances) downstream

-   Spread load across multiple downstream instances
-   Expose a single point of access (DNS) to your application
-   Seamlessly handle failures of downstream instances
-   Do regular health checks to your instances
-   Provide SSL termination (HTTPS) for your websites
-   Enforce stickiness with cookies
-   High availability across zones
-   Separate public traffic from private traffic
-   Health checks are done on the EC2 instances to know if instances it forwards traffic to are available to reply to requests
-   The health check is done on a port and a route (/health is common) If the response is not 200 (OK), then the instance is unhealthy

### Types of load balancers :

-   **Classic Load Balancer(depricated)**
    -   for HTTP, HTTPS, TCP, SSL (secure TCP)
-   **Application Load Balancer**
    -   for HTTP, HTTPS, WebSocket
    -   works on layers 7
    -   used for both VMs and containers
    -   support redirect http to https
    -   Routing
        -   routes based on path in URL
        -   routes based on hostname in URL
        -   routes based on Query String, Headers
    -   has a port matching feature to redirect to a dynamic port in ECS
    -   can use 1 ALB for multiple applications
    -   The application servers don’t see the IP of the client directly
        -   The true IP of the client is inserted in the header X-Forwarded-For
        -   We can also get Port (X-Forwarded-Port) and proto (X-Forwarded-Proto)
    -   Target Groups
        -   EC2 instances (can be managed by an Auto Scaling Group) – HTTP
        -   ECS tasks (managed by ECS itself) – HTTP
        -   Lambda functions – HTTP request is translated into a JSON event
        -   IP Addresses – must be private IPs
-   **Network Load Balancer**
    -   for TCP, TLS (secure TCP), UDP
    -   works on layer 4
    -   Forward TCP & UDP traffic to your instances
    -   ideal for Handling millions of request per seconds
    -   NLB has one static IP per AZ, and supports assigning Elastic IP(helpful for whitelisting specific IP)
    -   Target Groups
        -   EC2 instances
        -   IP Addresses – must be private IPs
        -   Application Load Balancer
-   **Gateway Load Balancer** - Operates at layer 3 (Network layer) – IP Protocol - it basically acts like a middle b/w your application and you that way you can perform certain action to the user before user get to the ALB - example : Firewalls, Intrusion detection and prevention systems, deep packet inspection systems, payload manipulation - basically used for security purposes - Uses the GENEVE protocol on port 6081 - Target Groups - EC2 instances - IP Addresses – must be private IPs
    > Load balancers can also be setup to only allow traffic that passes through the load balancer

**Sticky Sessions** : Implement stickiness so that the same client is always redirected to the same instance behind a load balancer

-   works for CLB, ALB, NLB
-   CLB & ALB uses a "cookie" to implement stickiness and has an expiration date that you control
-   Use case: so that the user don't lose his session data
-   may bring imbalance to the load over the backend EC2 instances
-   Sticky Session Cookie Names - Application based cookies - Custom cookie - Generated by the target - Can include any custom attributes required by the application - Cookie name must be specified individually for each target group - Don’t use AWSALB, AWSALBAPP, or AWSALBTG (reserved for use by the ELB) - Application cookie - Generated by the load balancer - Cookie name is AWSALBAPP - Duration-based Cookies - Cookie generated by the load balancer - Cookie name is AWSALB for ALB, AWSELB for CLB  
    **Cross-Zone Load Balancing** : Each load balancer instance distributes evenly across all registered instances in all AZ
-   ALB : enabled by default, can't be disabled
-   NLB : disabled by default, Charged
-   GLB : disabled by default, No charges

**SSL/TLS** :

-   An SSL Certificate allows traffic between your clients and your load balancer to be encrypted in transit.
-   SSL : Secure Sockets Layer - to encrypt connections
-   TLS : Transport Layer Security - to encrypt connections
-   Public SSL certificates are issued by Certificate Authorities (CA)

    -   SNI solves the problem of loading multiple SSL certificates onto one web server (to serve multiple websites)
    -   It’s a “newer” protocol, and requires the client to indicate the hostname of the target server in the initial SSL handshake, The server will then find the correct certificate, or return the default one

# `Amazon RDS Proxy`

A proxy is like a middleman between your application and a resource it needs to interact with, such as a database or another server. Instead of the application directly connecting to the resource, it connects to the proxy, and the proxy manages the connection on its behalf.

-   RDS Proxy is never publicly accessible (must be accessed from VPC)
-   RDS Proxy efficiently manages a pool of database connections and reuses them, reducing the overhead caused by frequent opening and closing of database connections.
-   It supports applications with unpredictable traffic spikes by efficiently handling thousands of concurrent connections.
-   RDS Proxy reduces failover times for Aurora and RDS databases by preserving connections during database failovers.
-   Integrates with AWS Secrets Manager to securely store and retrieve database credentials.
-   Supports IAM authentication for better access control.
-   For Aurora clusters, RDS Proxy can route queries to the appropriate instance (reader or writer), optimizing performance.
-   It helps lower latency for applications by maintaining active database connections.

>## DNS( Domain Name Server )
    - DNS translates domain names (e.g., example.com) into IP addresses (e.g., 192.0.2.1)
    - User enters domain → Check local cache → Query DNS resolver → Perform recursive lookup.
        - Recursive Lookup : 
            - Root server → Provides TLD server (e.g., .com).
            - TLD server → Provides authoritative server for domain.
            - Authoritative server → Provides final IP address.
    - Cached at multiple levels (browser, OS, resolver) to reduce lookup time.
        - TTL (Time to Live) controls cache expiry.
    - DNS Records:
    - A Record: Maps domain to IPv4.
    - AAAA Record: Maps domain to IPv6.
    - CNAME Record: Alias for another domain.
    - MX Record: Email server mapping.
    - NS Record (Name Server Record) : Specifies the authoritative name servers for the domain.


    - Key Benefits:
    - Scalability.
    - Fault tolerance.
    - Low latency through caching and distribution.
    - http://api.www.example.co : 
        - https:// : Protocol: Specifies how data is transmitted (HTTP in this case). Not part of DNS
        - api.www.example.co : Domain Name: Handled by DNS.
            - api → Subdomain (often used for APIs).
            - www → Subdomain (optional and can exist alongside others).
            - example → Second-level domain (unique identifier).
            - .co → Top-level domain (TLD, a country or general domain type).

# `Amazon Route 53`
Amazon Route 53 is a scalable and highly available Domain Name System (DNS) web service offered by AWS. It’s used for managing domain names, directing internet traffic, and ensuring applications remain highly available.
- Route 53 resolves domain names to IP addresses, enabling browsers to connect to web applications.
-  Supports public and private DNS.
- Allows you to purchase and manage domain names directly through AWS.
- Hosted Zones : 
    - Public Hosted Zones – contains records that specify how to route traffic on the Internet (public domain names)
    - Private Hosted Zones – contain records that specify how you route traffic within one or more VPCs (private domain names)
- TTL (Time To Live) in DNS records refers to the amount of time (in seconds) that a record is cached by DNS resolvers before they query the authoritative DNS server again for updated information. It directly impacts how often DNS changes propagate across the internet. Except for Alias records, TTL is mandatory for each DNS record.
- Alias record : An Alias record is a special type of DNS record supported by services like Amazon Route 53 that points a domain name to an AWS resource (like an S3 bucket, CloudFront distribution, or Elastic Load Balancer) or another domain name.
- CNAME : A CNAME (Canonical Name) record maps one domain name to another domain name (never directly to an IP address).
- ROUTING POLICIES : 
    - *Simple Routing Policy* : 
        - Directs traffic to a single resource
        - Single website with no complex routing needs.
        - Example: Pointing example.com to an EC2 instance.
        - Create a single record with the desired IP address or domain.
    - *Weighted Routing Policy* : 
        - Distributes traffic across multiple resources based on assigned weights. 
        - Load balancing across multiple servers.
        - Like Testing new features (e.g., routing 90% to the current version and 10% to a new version).
        - Assign weights to each resource.
        - *Example*: 70% traffic to server1.example.com and 30% to server2.example.com
    - *Latency Routing Policy* : 
        - Directs traffic to the resource with the lowest network latency for the user. 
        - Global applications requiring low-latency access.
        - Example: Routing users in Europe to a server in Germany and users in Asia to a server in Singapore.
        - Create records for each resource and associate them with AWS regions.
    - *Geolocation Routing Policy* : 
        - Routes traffic based on the geographic location of the user.
        - Regional content delivery or regulatory compliance.
        - Example: Users in the US are routed to us.example.com, and users in Europe to eu.example.com
        - Specify a location (continent, country, or state/province) for each record.
        - A default record can be configured for unmatched locations.
    - *Geoproximity Routing Policy (with Traffic Flow)* : 
        - Routes traffic based on the geographic location of the resource and optionally shifts traffic between locations.
        - Prioritize traffic to specific regions but allow spillover when capacity is exceeded.
        - Example: Shift 20% of traffic from Asia to the US.  
        - Requires Route 53 Traffic Flow.
        - Use geographic coordinates and bias settings to configure traffic shifting.
    - *Failover Routing Policy* : 
        - Automatically redirects traffic to a healthy resource when the primary resource is unhealthy.
        - High availability and disaster recovery setups.
        - Example: Primary resource in us-west-1 and a backup in us-east-1.
        - Configure a primary and a secondary (failover) record.
        - Enable health checks for failover detection.
    - *Multi-Value Answer Routing Policy* : 
        - Returns multiple IP addresses for a DNS query, improving redundancy and load balancing.
        - Distribute traffic across multiple servers and provide failover without a load balancer.
        - Example: Return all healthy IPs of web servers.
        - Enable health checks to ensure only healthy endpoints are returned.
    > IP-based routing refers to directing traffic based on the source IP address or destination IP address of a request. This type of routing is typically used in network configurations or DNS to deliver traffic to specific resources or services.

    > Multi-Value Routing is a routing policy in Amazon Route 53 that returns multiple healthy IP addresses in response to a DNS query. It provides basic load balancing and improves fault tolerance by letting clients choose from multiple resources.

# `Amazon Virtual Private Cloud (VPC)`
Amazon VPC is a foundational AWS service that allows you to create a logically isolated virtual network within the AWS cloud. It provides control over your virtual networking environment, enabling you to define your network configuration, select IP ranges, and create subnets.

### Subnets : 

Subdivision of the VPC IP address range.
- Public Subnets: Accessible from the internet (e.g., for web servers).
- Private Subnets: Not directly accessible from the internet (e.g., for databases).
# `Internet Gateway (IGW)`
An Internet Gateway allows communication between resources in your VPC and the internet. It's a horizontally scaled, redundant, and highly available component that can be attached to your VPC.
- *For public subnet internet connection*
- The Internet Gateway connects your public subnet to the internet, enabling instances like web servers to communicate with external clients (browsers, API calls, etc.).
- Route Table Configuration: Once an Internet Gateway is attached to a VPC, you'll update the route table for the public subnet(s) to route all outbound traffic (0.0.0.0/0) to the IGW.
# `NAT Gateway/Instance` 
A NAT (Network Address Translation) Gateway or NAT Instance allows instances in a private subnet to access the internet without exposing those instances to direct incoming traffic.
- *For private subnet internet connection*
> **Differences Between NAT Gateway and NAT Instance** :
- NAT Gateway: A fully managed service by AWS that automatically scales to accommodate traffic. It is more resilient and highly available.
- NAT Instance: A manually configured EC2 instance that serves as a NAT device. While it can be cheaper for low-traffic use cases, it's less scalable and resilient than the NAT Gateway.
- Private subnet instances need to access the internet (for example, to download software updates). You configure the route table of the private subnet to route outbound traffic to the NAT Gateway (or NAT Instance), which has internet access.
### Route Tables: 

A Route Table is a set of rules (routes) that determine how traffic should be directed in your VPC. Each subnet in your VPC is associated with a route table, and you can either use the default route table or create custom ones.
- Destination: The destination IP address or range of IPs for which the rule applies (e.g., 0.0.0.0/0 for all internet traffic).
- Target: The resource or gateway that traffic is sent to (e.g., Internet Gateway, NAT Gateway, EC2 instance).
- In a public subnet, route 0.0.0.0/0 to the Internet Gateway.
- In a private subnet, route 0.0.0.0/0 to the NAT Gateway.
### Security Groups : 

A Security Group is a virtual firewall for controlling inbound and outbound traffic to AWS resources, such as EC2 instances.
- Stateful: If you allow inbound traffic on a specific port, the return traffic is automatically allowed, even if you don't explicitly define outbound rules.
- Instance Level: Security groups are associated with EC2 instances and control access to them.
- Allow HTTP (port 80) and HTTPS (port 443) traffic to your web server.
- Deny inbound SSH (port 22) traffic from all sources except a specific IP range.
### Network Access Control Lists (NACLs): 

A Network Access Control List (NACL) is an additional layer of security that controls traffic at the subnet level, unlike security groups, which are instance-level firewalls.
- Stateless: You need to explicitly define both inbound and outbound rules. If you allow inbound traffic, you also need to specify the return traffic.
- Subnet Level: NACLs are applied to subnets and can control traffic entering or leaving the subnet.
- NACLs can be used for an extra layer of security, for instance, blocking specific IP addresses or ranges from accessing any resource within a subnet.
# `Elastic IP (EIP)`

An Elastic IP (EIP) is a static, public IPv4 address that can be associated with your EC2 instance or NAT Gateway. It is used when you need a fixed public IP address that doesn’t change even if you stop and restart your instance.
- When you associate an Elastic IP with an EC2 instance, it becomes the public IP address of that instance. If the instance is stopped or restarted, the IP remains the same as long as you don’t disassociate it.
- You want to maintain a persistent IP for a service (e.g., a web application) that needs to be accessible from the internet.
# `VPC Peering` 
VPC Peering allows two VPCs to communicate as if they are in the same network, even if they are in different regions or accounts.
- Traffic Routing: You configure route tables in each VPC to allow traffic to flow between them.
- Limitations: A VPC peering connection is one-to-one (i.e., one VPC can only be peered with one other VPC at a time).
- If you have multiple VPCs (e.g., a VPC for staging and one for production), you can peer them to enable secure communication between resources in these environments.
# `VPC Endpoints` 
A VPC Endpoint allows private, secure access to AWS services (like S3, DynamoDB) from within your VPC without using the internet or a NAT device.
- Interface Endpoints: Used for services that are accessed via IP addresses, like AWS APIs.
- Gateway Endpoints: Used for services like Amazon S3 and DynamoDB, where traffic flows through the AWS network.
- Access S3 buckets from private subnets without going over the internet (improving security and latency).
# `AWS Transit Gateway`
Simplifies VPC-to-VPC communication in large-scale environments by acting as a central hub. It can also integrate with on-premises data centers, enabling easier management of multiple VPCs.
# `Direct Connect`
Establishes a dedicated network connection between your on-premises infrastructure and AWS. It provides low latency, higher bandwidth, and secure private connections to AWS resources.
**Public and slow because its a network connection**
# `VPN Connection`
Virtual Private Network (VPN) connection enables secure communication between your on-premises network and your VPC over the public internet.

# `Site-to-Site VPN`
A Site-to-Site VPN is a connection between two networks over the internet, such as connecting your on-premises data center or branch office network to your AWS VPC. This allows the on-premises network and the VPC to communicate securely as if they were part of the same local network, using IPsec (Internet Protocol Security) encryption to secure the data traffic.
**Private and fast because its a physical connection**

### Amazon S3 – Security

### Storage Classes : 
- Standard: Frequently accessed data.
- Standard IA - Infrequently accessed data
- Intelligent-Tiering: Data with unknown access patterns.
- One Zone-IA: Infrequently accessed data that doesn't need to be in multiple Availability Zones.
- Glacier Instant Archive : For archival data that can be retrieved in millisecond.
- Glacier Flexible Archive : Flexible retrieval time.
- Glacier Deep Archive: Lowest-cost storage for data that is rarely accessed.
- S3 can host static websites and have them accessible on the Internet
- *Versioning*: S3 allows you to keep multiple versions of an object. When versioning is enabled, each change to an object creates a new version, so you can retrieve or restore any previous version.
- Amazon S3 – Replication (CRR & SRR)
    - Must enable Versioning in source and destination buckets
    - Cross-Region Replication (CRR)
    - Same-Region Replication (SRR)
    - Buckets can be in different AWS accounts
    - Copying is asynchronous
    - Must give proper IAM permissions to S3
    - Use cases:
        - CRR : compliance, lower latency access, replication across accounts
        - SRR : log aggregation, live replication between production and test accounts
    - After you enable Replication, only new objects are replicated
    - Optionally, you can replicate existing objects using S3 Batch Replication, Replicates existing objects and objects that failed replication
    - For DELETE operations :
        - Can replicate delete markers from source to target (optional setting)
        - Deletions with a version ID are not replicated (to avoid malicious deletes)
    - There is no “chaining” of replication :
        - If bucket 1 has replication into bucket 2, which has replication into bucket 3
        - Then objects created in bucket 1 are not replicated to bucket 3
- *Lifecycle Policies*: You can set rules for automatic transition of objects to different storage classes or deletion after a specified period. This helps manage costs.
- *Encryption*: S3 supports both server-side encryption (SSE) and client-side encryption to protect data at rest. Encryption options include SSE-S3, SSE-KMS (AWS Key Management Service), and SSE-C (customer-provided keys).
- *Cross-Region Replication (CRR)*: CRR allows you to automatically replicate objects across different AWS regions for disaster recovery or compliance reasons.
- *Event Notifications*: You can configure event notifications to trigger Lambda functions, SQS queues, or SNS topics based on specific actions (like object uploads or deletions).
- *S3 Transfer Acceleration*: Speeds up the upload and download of data by routing traffic through AWS edge locations.
- *Static Website Hosting*: S3 can host static websites, serving HTML, CSS, JavaScript, and media files.
- *Multipart Upload*: For large objects, you can upload parts of the object in parallel and then combine them into a single object to improve efficiency and reduce errors.
- *Logging and Monitoring*: S3 supports logging access requests and integrates with AWS CloudTrail and Amazon CloudWatch for monitoring and auditing S3 activity.
- *Pre-signed url* : 
A pre-signed URL is a URL that grants temporary access to a private object stored in Amazon S3
- Amazon S3 Access Points are a feature of Amazon S3 that simplify managing access to shared data in S3. They provide a unique DNS-compliant hostname that acts as a point of entry to a specific S3 bucket, simplifying access control when multiple applications or users need to access data in the same bucket.
- Amazon S3 Object Lambda allows you to add custom processing to the data retrieved from Amazon S3 before it is returned to an application or user. With S3 Object Lambda, you can modify the content of an S3 object (such as applying transformations or filtering data) on-the-fly when it is requested, without modifying the underlying S3 object.
